### Local tests
```
# Execute test locally
# 1. Bump chart version in Chart.yaml
# 2. Publish Chart ot test repository
build/publishChart.sh test
# 3. Update chart version in test deployment of bip
# 4. Login to Openshift
# 5. Run tests
test/executeTestPipelineRuns.sh -m local -c build|all
```

### Tests through sti-hcp pipeline
```
# 1. Bump chart version in Chart.yaml to required release version of the chart. Build version will be automatically generated by the sti-hcp pipeline run
# 2. Git commit and push your changes
# 3. Tests will be automatically executed by sti-hcp pipelinerun
```

#### Monorepo
##### Why?
The general-application-pipeline has a monorepo feature which is used by several teams and therefore needs to be tested throroughly.

##### What?
The monorepo test consists of four different test cases:
* [Application with specified context path (app1-with-context-paths)](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example-gitops/browse/pipeline/values.yaml?at=refs%2Fheads%2Fmonorepo-support#19-28)
* [Application without context path (app2)](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example-gitops/browse/pipeline/values.yaml?at=refs%2Fheads%2Fmonorepo-support#30-38)
* [Application pointing to a non-existing git repository (app3)](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example-gitops/browse/pipeline/values.yaml?at=refs%2Fheads%2Fmonorepo-support#40-47)
* [Application from existing repo with incorrect container file path (app1-with-context-paths)](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example-gitops/browse/pipeline/values.yaml?at=refs%2Fheads%2Fmonorepo-support#50-58)


Containerfiles and the build setup of the test application used in test cases `app1-with-context-paths` and `app2` can be found [here](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example/browse?at=develop-monorepo).
The relevant GitOps configuration for this test can be found [here](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example-gitops/browse/pipeline?at=refs%2Fheads%2Fmonorepo-support).

##### How?

You should be aware that this is a manual test case and needs a (human) trigger in order to get executed.

The monorepo setup incorporates a piece of software which checks if the application's buildconfig (`Containerfile`) changed. The pipeline is only executed for apps which have changes in their `Containerfile`. 

You need to check out the [sti-gap-example](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example/browse?at=refs%2Fheads%2Fdevelop-monorepo) repository on the `develop-monorepo` branch. In case you need to trigger this test for apps `app1-with-context-paths` and `app2` you'd need to perform changes in their `Containerfile`s (e.g. add line breaks at the end of the file), commit and push them to `develop-monorepo`.

After the pipeline run has been triggered you should be able to observe them here:
* [app1-with-context-paths](https://console-openshift-console.apps.p-szb-ros-shrd-npr-01.cloud.admin.ch/k8s/ns/bit-amber-pipelines-d/tekton.dev~v1~Pipeline/app1-with-context-paths-gp-build-pipeline/Runs)
* [app2](https://console-openshift-console.apps.p-szb-ros-shrd-npr-01.cloud.admin.ch/k8s/ns/bit-amber-pipelines-d/tekton.dev~v1~Pipeline/app2-gp-build-pipeline/Runs)

#### Run sca-check with custom Dependency-Track
In order to run the sca-check and upload the results to a custom Dependency-Track instance, a few things are required

* A running Dependency-Track instance (apiserver and frontend with internal DB)
  1. Run `oc apply -f ./dependency-track-helm/dependency-track-deployment.yaml`
  2. Login to the frontend with admin:admin and change the dummy pw
  3. Get the API Key: go to Administration->Access Management->Teams->(select a team name and generate the API Key)
* A property in MAV containing the API Key of your Dependency-Track instance
  * Add a key in your pipelines-d namespace (i.e. "dependency-track-test-api-key" under "default") and set the value to the API key (be aware the key has to be replaced regularly)
* A pipeline running the sca-check ([gap-add-dt-task-test](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-pipelines-gitops/browse/p-szb-ros-shrd-npr-amber/applicationset.yaml))
  * Edit the file [values.add-dt-task.yaml](test/values.add-dt-task.yaml) to test different constellations
* A Containerfile which is built with the pipeline [sti-gap-example on branch develop-add-dt-task](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example/browse?at=develop-add-dt-task)
  * A webhook is configured which triggers the event listener to start a PipelineRun

In order to deprovision the test set up
1. Run `oc delete -f dependency-track-helm/dependency-track-deployment.yaml`
2. Delete your key in MAV

#### Run malware-scan with Antivirus Service
A prerequisite to use or test the malware-scan is an antivirus service ordered on MyAtlantica service portal:
1. Go to the Marketplace on MyAtlantica
2. Open the Container Platform for your cluster
3. Select Antivirus and fill out the form
4. Open the `Abonnement Detail+` of your Antivirus service and go to the tab Service Bindings
5. Click on `+` to create a new binding
6. The account type has to be `service-account` in case you want to use the binding in the pipeline
7. Configure your application as in [values.add-malware-scan-task.yaml](test/values.add-malware-scan-task.yaml) with the values of your antivirus service

## Release process

* Starting from a branch like `STI-1234-my-story-title`
  * Update [CHANGELOG.md](CHANGELOG.md) + [README.md](README.md) with `helm-docs .`
  * Bump Version of Chart [Chart.yaml](Chart.yaml)
* Depending on context, either
  * Create a release branch `release/gap-v4`
    * Create PR to `release/gap-v4`
      * Build notification must be successful
      * We will not do a review to each individual changes when on a release branch, we will focus on a single global big review for the release when merging to `master`
      * Merge it
  * Create PR to `master`
    * Build notification must be successful
    * Must be reviewed
* When merged to `master`, the pipeline's Helm chart is available
* Update related other project to reflect current latest state, especially breaking changes
  * [bit-example-renovate-gitops](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/bit-example-renovate-gitops): is provided as an example for Renovate integration with GAP
    * Version in [chart](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/bit-example-renovate-gitops/browse/pipeline/Chart.yaml#10)
    * Corresponding [values](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/bit-example-renovate-gitops/browse/pipeline/values.yaml)
  * [sti-gap-example-gitops](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example-gitops): is provided as example for GAP integration in Openshift
    * Version in [chart](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example-gitops/browse/pipeline/Chart.yaml#10?at=refs%2Fheads%2Fprd)
    * Corresponding [values](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-gap-example-gitops/browse/pipeline/values.yaml?at=refs%2Fheads%2Fprd)

### Development in progress release process

Purpose of this release process is only for "development in progress" and not for "official releases".
It spares the need to commit to Bitbucket to trigger webhook and [HCP](https://bitbucket.bit.admin.ch/projects/STIRHOS/repos/sti-hcp/browse).

1. Update [CHANGELOG.md](CHANGELOG.md) + [README.md](README.md)
2. Bump Version of Chart [Chart.yaml](Chart.yaml)
3. Run `./build/publishChart.sh test` from root of this Chart (example /home/dev/development/generic-application-pipeline)
4. Go into the ArgoCD application, which you want to test your new Chart and make a sync
5. Go into the [Openshift Console](https://console-openshift-console.apps.ccp11-e.atlantica.admin.ch/) and trigger the Pipeline
6. If it's green, upload the new Chart `./build/publishChart.sh prod`
